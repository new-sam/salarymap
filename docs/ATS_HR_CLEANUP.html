04DB — 살아있는 것

객체역할상태
recruiter_companies (table)회사 명부 (id, name, email_domain, tax_id, verified_at, created_by)유지
recruiter_users (table)회사 담당자 (user_id ↔ company_id ↔ role)유지
jobs.company_id (column)jobs ↔ recruiter_companies 연결유지
jobs.status (column)draft / pending_review / live / paused / closed유지
recruiter_jobs (VIEW)jobs + recruiter_companies 조인 view유지
job_applications (table) + RLS지원자 + 4단계 status (기존 활용, RLS 신규 추가)유지

05DB — 운영에 잔재할 수 있는 것

코드는 머지로 정리됐지만 운영 Supabase의 테이블·컬럼·row 는 별도. 아래는 운영 DB 상태에 따라 처리 필요.

객체역할처리
hr_users (table)옛 HR 사용자 명부DROP
candidates (table)HR 인재 풀 — user_profiles와 중복 가능데이터 확인 후 DROP
talent_profile 관련 테이블HR 인재 프로필DROP
user_profiles.role 값이 'hr' 인 rowHR로 가입했던 사용자개수 확인 → 'seeker'로 reset or 그대로 두기
supabase/migrations/20260512_hr_onboarding.sqlhr_users 생성 마이그레이션 파일머지 시 함께 삭제됐는지 확인
supabase/migrations/20260512_candidates.sqlcandidates 테이블 생성동일
supabase/migrations/20260512_talent_*talent 관련동일

06결정 포인트

① candidates 운영 DB 에 데이터가 있는지 (있으면 마이그레이션 절차 필요)
② user_profiles.role이 'hr'인 row 처리 방법 — 그냥 둘지, 'seeker'로 reset
③ HR 도메인에서 production에 실제 가입한 사용자가 있는지
④ DB DROP 시점 — 코드 머지 후 1주 안전 기간 후 DROP vs 즉시 DROP
⑤ recruiter_* 테이블에 production 시드 데이터 필요 여부 (테스트용 회사 row)
